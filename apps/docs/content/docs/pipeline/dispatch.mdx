---
title: "Stage 3: Dispatch"
description: Send selected candidates to Devin AI agents for full curation.
---

Each selected GHSA is dispatched to a [Devin](https://devin.ai/) agent. The agent follows a structured prompt that walks it through the full curation workflow. Each agent opens a pull request containing exactly two files, which is then reviewed before merging.

## Usage

```bash
uv run python -m pipeline.dispatch_devin --count 10
```

| Flag | Default | Description |
| --- | --- | --- |
| `--input` | `output/candidates.jsonl` | Input from Stage 2 |
| `--count` | (required) | Number of candidates to dispatch |
| `--offset` | 0 | Skip first N candidates (for batched dispatch) |
| `--branch` | (none) | Shared branch name (all agents commit here instead of per-task branches) |
| `--dry-run` | off | Render prompts without calling the API |
| `--delay` | 2.0 | Seconds between API calls |

### Batched Dispatch

For large runs, use `--offset` to dispatch in batches:

```bash
# Dispatch candidates 0–9
uv run python -m pipeline.dispatch_devin --count 10 --offset 0

# Dispatch candidates 10–19
uv run python -m pipeline.dispatch_devin --count 10 --offset 10
```

### Shared Branch Mode

By default, each agent creates its own branch (`curate/{task_id}`). Use `--branch` to have all agents commit to a single shared branch:

```bash
uv run python -m pipeline.dispatch_devin --count 10 --branch curate/batch-001
```

Agents in shared branch mode will not clobber each other's commits — each pulls the latest changes before committing.

## Per-Advisory Workflow

Each Devin agent executes the following steps:

1. **Read the advisory** — Navigate to the GHSA URL, extract description and references.
2. **Resolve the patch commit** — Find the patch commit SHA from references (may involve resolving a PR merge commit or a release tag).
3. **Identify the pre-patch commit** — First parent of the patch commit. This is the commit served to agents during evaluation.
4. **Examine the patch diff** — Identify modified source files and functions (excluding tests, docs, configs, changelogs). Flag noisy patches (>3 non-test files).
5. **Classify the vulnerability** — Verify the pre-assigned vulnerability class against the diff. Report mismatches rather than silently overriding.
6. **Derive locations** — Identify file(s) and function(s) where the vulnerability exists in the pre-patch code. Priority: advisory description > patch diff > semantic analysis.
7. **Write the L1 localization hint** — Vague area hint pointing at a broad codebase region without revealing the vulnerability type or specific files/functions.
8. **Write the L2 CVE hint** — Scrubbed advisory description with all file paths, function names, line numbers, variable names, and code snippets removed.
9. **Generate the task ID** — Format: `ecvebench-{project}-{NNN}`.
10. **Create the task file** — `benchmark/data/tasks/{task_id}.json`.
11. **Create the metadata file** — `benchmark/internal/metadata/{GHSA_ID}.json`.
12. **Open a PR** — Branch: `curate/{task_id}` (or the shared branch if `--branch` was used).

## Output Artifacts

Each curated advisory produces two files:

- **Task file** (`data/tasks/{task_id}.json`) — Contains codebase pointer, hints at each difficulty, and ground truth. See [Task Format](/docs/benchmark/task-format).
- **Metadata file** (`internal/metadata/{GHSA_ID}.json`) — Contains post-patch commit, curation notes, dataset version (not exposed to agents).

## Prompt Template

The dispatch script populates the prompt template at `benchmark/docs/prompts/curation_agent.md` with pre-computed fields from the candidate JSONL (vulnerability class, CVSS, CVE ID, CWE IDs, ecosystem, snapshot date).

## Environment Variables

Required in `.env` (see `.env.example`):

| Variable | Description |
| --- | --- |
| `DEVIN_API_KEY` | Devin API key |
| `DEVIN_ORG_ID` | Devin organization ID |
| `DEVIN_USER_ID` | Devin user ID for session attribution |
| `DEVIN_REPO` | Target repository (e.g. `owner/repo`) |
